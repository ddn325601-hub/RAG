## 3. PPT 截图内容建议

### 3.1 封面

建议显示：

- 项目名称：智能 OnCall Agent
- 副标题：面向客服、售后与运维场景的知识增强问答智能体
- 关键词：RAG、Agent、DashScope、Milvus、Spring Boot

### 3.2 选题意义

建议显示：

- 一线客服需要快速回答用户问题。
- 售后人员需要准确解释订单、退款、政策和流程。
- 运维人员需要在告警出现后快速定位日志、指标和处理步骤。
- 传统处理方式依赖人工经验，响应速度和一致性不足。

### 3.3 业务痛点

建议显示三类痛点：

- 资料分散：FAQ、手册、日志说明、告警流程分散在多个文件。
- 响应慢：一线人员需要反复检索文档和询问技术同事。
- 不可追溯：普通大模型回答缺少来源，难以用于严肃业务场景。

### 3.4 系统架构

建议显示链路：

```text
前端页面 -> Spring Boot 后端 -> RAG 检索 -> Milvus 向量库
                         -> DashScope 大模型
                         -> Agent 工具调用
                         -> /chat 标准接口
                         -> ECS + Nginx 部署
```

讲解重点：

- 上传资料后进行分片和向量化。
- 问答时先检索知识库，再由模型组织答案。
- 前端展示引用来源，保证答案可追溯。
- `/chat` 接口支持 Token 鉴权，适合外部系统调用。

### 3.5 创新与特色

建议显示：

- 知识增强：支持企业资料投喂和 TopK 检索。
- 智能体化：结合文档检索、日志、指标等 OnCall 工具思路。
- 可追溯：回答展示引用来源、片段和相似度。
- 可部署：已完成本地运行、云端部署、接口鉴权和演示材料。

### 3.6 结果与分析

建议显示：

- 已投喂 7 份客服、售后、运维资料。
- 支付订单超时问题可返回结构化排查步骤。
- 引用来源显示 TopK=3。
- `/chat` 无 Token 返回 401，带 Token 返回 200。

### 3.7 应用价值

建议显示：

- 客服：缩短标准问题答复时间。
- 售后：提高订单、退款、政策类问题处理一致性。
- 运维：沉淀告警处理流程，辅助新人快速上手。
- 企业知识管理：把分散资料转化为可检索、可追溯的问答能力。

## 4. 剪映操作建议

1. 建立 1920 x 1080 横屏项目。
2. 先导入 PPT 截图，再导入 `OnCallAgent_Function_Demo.mp4`。
3. PPT 截图每页使用 10 到 25 秒，不要做复杂转场，使用淡入淡出即可。
4. 网页实操视频保留原始画面，不叠加说明框、箭头、红线或水印。
5. 导入字幕文件，优先使用 `.srt`；如果要更精细控制样式，可使用 `.ass`。
6. 字幕建议放在底部偏上位置，避免遮挡网页输入框和引用来源。
7. 背景音乐可选很轻的科技感音乐，音量控制在 8% 到 12%，不要盖过旁白。
8. 导出参数建议：MP4、H.264、1080p、30fps、码率 8Mbps 左右。

## 5. 完整旁白稿

大家好，我们的作品是智能 OnCall Agent，面向客服、售后和运维场景，目标是把分散的业务资料转化为可检索、可追溯、可调用的智能问答能力。

在真实业务中，一线人员经常需要同时查 FAQ、产品说明、售后政策、故障手册、日志说明和告警流程。资料分散会导致响应慢，经验依赖强，也让回答难以复核。

本项目希望解决三个问题：第一，让业务资料可以被统一投喂和检索；第二，让大模型回答时有明确来源；第三，让问答能力可以通过标准接口接入外部系统。

系统架构上，前端负责聊天、知识库管理、引用来源和接口调试；后端使用 Spring Boot 提供问答、上传、检索和比赛接口；文档上传后会被分片、向量化，并写入 Milvus。

问答时，系统先从知识库召回相关片段，再交给 DashScope 模型和智能体流程组织答案。云端部署部分通过 ECS、Nginx 和 Java 服务完成，方便评审或外部系统访问。

项目的主要特色是知识增强、来源可追溯、接口可鉴权和场景可扩展。它不是只展示一个聊天框，而是围绕客服、售后、运维的真实资料处理流程构建。

接下来进入系统页面。左侧可以看到智能问答、知识库管理和接口调试三个入口。首页保留真实产品交互，不展示临时说明卡片，便于直接作为产品演示素材。

在知识库管理页面，可以看到系统已经投喂了 7 份资料，包括支付订单排查手册、售后政策、运维告警流程、日志指标说明、图片工单分诊、客服 FAQ 和产品支持总览。

这里的上传入口用于继续投喂 Markdown 或 TXT 资料。资料上传后会进入解析、切片、向量化和入库流程，后续问答会基于这些资料召回证据。

下面演示一个典型问题：支付订单超时如何排查？这个问题同时涉及支付回调、订单服务、消息队列、库存、风控和数据库等链路，适合验证 RAG 能力。

回答生成完成后，页面会从本次回答顶部开始展示。可以看到系统先给出适用场景和标准排查步骤，而不是直接跳到最终结论。

继续向下可以看到具体排查过程，包括确认支付结果真实性、搜索应用日志、检查订单服务核心日志流，以及验证消息中间件健康度。

在处理建议部分，系统给出冻结后续支付请求、同步通知技术支持、执行补单或退款、记录完整时间线等操作建议，便于一线人员直接执行。

回答末尾展示引用来源，当前 TopK 为 3。每条来源包含文档名称、片段标题、摘要和 score，这说明回答不是凭空生成，而是可以追溯到知识库证据。

最后展示接口调试。无 Token 调用 `/chat` 会返回 401，带正确 Token 调用会返回 200 和正常问答结果，说明项目具备基本的接口安全控制。

总体来看，本项目已经完成真实模型接入、RAG 知识库、引用来源展示、知识库管理、接口鉴权和云端部署。后续可以继续接入真实工单、监控平台和自动化评测集，扩展为企业级智能 OnCall 助手。
